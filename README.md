# ГЭУ Диалог

Дипломный проект: веб-приложение для приемной комиссии СПбГЭУ. Система работает как чат-бот для абитуриентов, умеет проверять позиции абитуриента в конкурсных списках, сохранять проверки сотрудников и анализировать ошибки ответов агента по базе источников.

Публичная версия: [https://geu-dialog.onrender.com](https://geu-dialog.onrender.com)

## Что умеет проект

- Отвечает на типовые вопросы абитуриентов.
- Распознает код абитуриента и ищет его в подключенных конкурсных списках.
- Загружает конкурсные списки через серверный proxy, чтобы обойти ограничения браузера и CORS.
- Показывает найденные позиции абитуриента по конкурсным группам.
- Сохраняет историю диалогов и журнал проверок сотрудника.
- Позволяет вручную проверить ответ чат-бота через форму `Тест анализа`.
- Выполняет поиск релевантных источников по правильному ответу сотрудника.
- Сравнивает ответ бота, источник бота и правильный ответ сотрудника.
- Использует LLM для связного объяснения причины ошибки.
- Выводит доказательства в виде текстовых фрагментов, а не только названий файлов и страниц.

## Основные сценарии

### 1. Абитуриент спрашивает у бота

Пользователь открывает сайт, задает вопрос или вводит код абитуриента. Если введен код, приложение ищет его в конкурсных списках. Если введен обычный вопрос, бот отвечает по локальной базе знаний.

### 2. Сотрудник проверяет ответ

Сотрудник может отметить ответ как правильный или неправильный, добавить корректный ответ и сохранить проверку в журнал.

### 3. Ручной тест анализа ошибки

В панели `Тест анализа` вручную вводятся:

1. вопрос абитуриента;
2. ответ чат-бота;
3. фрагмент источника, на который опирался чат-бот;
4. правильный ответ сотрудника.

После этого агент:

1. ищет top-k источники по правильному ответу сотрудника;
2. сравнивает найденные источники с источником ответа бота;
3. определяет тип ошибки;
4. вызывает LLM для нормального текстового объяснения;
5. показывает причину, рекомендацию, top-k фрагменты и доказательства.

## Архитектура

```text
Браузер пользователя
        |
        | index.html / style.css / script.js
        v
Node.js server.js
        |
        |-- static files
        |-- SQLite: data/dialogs.sqlite
        |-- proxy к priem.unecon.ru
        |-- /api/source-search
        |-- /api/error-analysis
        v
error_analysis_agent.js
        |
        |-- data/combined_data.json
        |-- TF-IDF + cosine similarity
        |-- local rules
        |-- LLM: pollinations / openai-fast
```

## Требования

- Node.js `>= 22.19.0`
- Браузер
- Для полной работы анализа нужен файл `data/combined_data.json`

Проект не требует сборки фронтенда: интерфейс написан на обычных `HTML`, `CSS`, `JavaScript`.

## Переменные окружения

### Основные

| Переменная | Назначение | Значение по умолчанию |
|---|---|---|
| `PORT` | порт сервера | `3000` локально, на Render задается платформой |
| `HOST` | адрес прослушивания | `0.0.0.0` |
| `SOURCE_DATA_PATH` | путь к `combined_data.json`, если файл лежит не в `data/` | пусто |

### LLM

На Render используется:

```text
LLM_PROVIDER=pollinations
LLM_MODEL=openai-fast
LLM_TIMEOUT_MS=120000
LLM_MAX_TOKENS=1000
```

Для OpenAI-compatible API можно задать:

```powershell
$env:LLM_ENDPOINT = "https://api.openai.com/v1/chat/completions"
$env:LLM_API_KEY = "ваш_api_ключ"
$env:LLM_MODEL = "gpt-4o-mini"
```

Если LLM недоступна, агент использует локальный fallback: поиск и правила анализа остаются рабочими, но объяснение будет менее гибким.

## API

Проверяет, что сервер работает.

### `GET /api/state`

Возвращает сохраненные диалоги, сообщения и проверки из SQLite.

Импортирует состояние из старого локального хранилища браузера в SQLite.

Создает или обновляет чат.

### `POST /api/chats/:id/messages`

Сохраняет сообщения конкретного чата.

Удаляет чат, его сообщения и связанные проверки.

### `GET /api/reviews`

Возвращает журнал проверок сотрудника.

### `POST /api/reviews`

Сохраняет проверку ответа бота.

Серверный proxy для загрузки конкурсного списка с `priem.unecon.ru`.

Нужен потому, что браузер не всегда может напрямую загрузить таблицы с внешнего сайта из-за CORS и сетевых ограничений.

### `POST /api/source-search`

Ищет top-k источники по тексту.

Пример:

```json

  "topK": 5
}
```

Ответ содержит найденные фрагменты:

```json
{
  "sources": [
    {
      "id": "source-367",
      "score": 0.2558,
      "fileName": "general_faq.json",
      "pageNumber": "3",
      "preview": "### Есть ли военная кафедра? Нет, но есть программа военной подготовки..."

  ],
  "stats": {
    "textSources": 723
  }
}
```

Запускает полный анализ ошибки.

Пример запроса:

## Как работает парсер конкурсных списков

Логика находится в `script.js` и `server.js`.

1. Пользователь вводит код абитуриента.
2. Фронтенд распознает числовой код.
3. Приложение получает список конкурсных групп.
4. Для каждой конкурсной группы формируется URL на `priem.unecon.ru`.
5. HTML конкурсного списка загружается через `/api/admission-list`.
6. Фронтенд разбирает таблицы в HTML.
7. В строках таблицы ищется код абитуриента.
8. Если код найден, система определяет конкурсную группу, форму обучения, основу, раздел списка и позицию.
9. Ответ форматируется для пользователя.

Для ускорения используется ограничение параллельности: списки проверяются не все сразу, а пачками.

Если загрузка внешнего сайта не удалась, бот сообщает, что конкурсные списки не удалось получить, и предлагает проверить запуск `server.js`.

## Как работает агент анализа ошибок

Главный файл: `error_analysis_agent.js`.

На вход подается:

1. `question` - вопрос абитуриента;
2. `agentAnswer` - ответ бота;
3. `agentSources` - фрагмент источника, на который опирался бот;
4. `adminAnswer` - правильный ответ сотрудника.

Дальше выполняются шаги:

1. Загружается `data/combined_data.json`.
2. Из базы исключаются записи с `"label": "TABLE"`.
3. Текстовые источники превращаются в TF-IDF векторы.
4. По `adminAnswer` выполняется поиск top-k источников.
5. Одинаковые найденные фрагменты удаляются из top-k.
6. Найденные источники сравниваются с `agentSources`.
7. Ответ бота сравнивается с правильным ответом сотрудника.
8. Локальные правила определяют базовый тип ошибки.
9. LLM получает вопрос, ответ бота, источник бота, правильный ответ, top-k и базовый анализ.
10. LLM формирует связное объяснение ошибки.
11. Backend нормализует результат, чтобы в доказательствах были текстовые фрагменты, а не только файл или страница.

### Типы ошибок

| Тип | Значение |
|---|---|
| `wrong-source` | неправильно выбран источник |
| `merged-rows` | объединены несколько строк или фрагментов |
| `nonexistent-direction` | появились лишние строки, направления или приписки |
| `incomplete-answer` | ответ неполный |
| `missing-agent-source` | источник агента не сохранен |
| `missing-admin-answer` | нет правильного ответа сотрудника |
| `needs-review` | нужна ручная проверка |

### Пример связного объяснения LLM

Если бот ответил:

```text
военная кафедра предоставляется лишь студентам аспирантам
```

а переданный источник был:

```text
военная кафедра предоставляется всем студентам очной формы обучения
```

агент должен объяснить:

```text
Агент ответил, что военная кафедра предоставляется лишь студентам аспирантам, однако в переданном фрагменте указано, что она предоставляется всем студентам очной формы обучения. Эталонный ответ сотрудника совпадает с этим фрагментом. Таким образом, агент добавил недопустимое ограничение, которое не содержится в источнике.
```

## Векторный поиск

В проекте используется локальный поиск, без отдельной внешней векторной базы.

Алгоритм:

1. текст нормализуется;
2. выделяются токены;
3. считаются частоты слов;
4. строится TF-IDF вектор;
5. для запроса строится такой же вектор;
6. похожесть считается через cosine similarity;
7. дополнительно учитывается пересечение токенов;
8. результаты сортируются по score;
9. дубли одинаковых фрагментов удаляются.

Это значит, что другой пользователь не обязан иметь свою базу или локальный поиск. На публичном сайте все выполняется на Render:

```text
браузер пользователя -> Render -> server.js -> combined_data.json -> результат
```

## База данных

Для истории диалогов и журнала проверок используется SQLite:

```text
data/dialogs.sqlite
```

Таблицы:

- `chats` - список диалогов;
- `messages` - сообщения внутри диалогов;
- `reviews` - проверки ответов сотрудником.

Важно: на бесплатном Render SQLite может сбрасываться при перезапуске сервиса, потому что файловая система не предназначена для постоянного хранения. Для дипломной демонстрации это допустимо. Для продакшена лучше заменить SQLite на Postgres.

## Частые проблемы

### Сайт открывается, но конкурсные списки не загружаются

Проверить:

- запущен ли `server.js`;
- работает ли `/api/health`;
- доступен ли `priem.unecon.ru`;
- не изменилась ли структура HTML таблиц на сайте приемной комиссии.

### На локальном компьютере код абитуриента находится, а у другого пользователя нет

Если пользователь открывает `localhost`, он обращается к своему компьютеру, а не к вашему. Для внешнего доступа нужно открывать публичную ссылку Render:

```text
https://geu-dialog.onrender.com
```

### Cloudflare Tunnel Error 1033

Это означает, что туннель `cloudflared` не работает или компьютер выключен. Решение - использовать деплой на Render, тогда сайт не зависит от включенного компьютера.

### Агент пишет fallback вместо LLM

Причины:

- LLM-сервис временно не ответил;
- не заданы переменные окружения;
- истек timeout;
- внешний сервис вернул ошибку.

При этом локальный анализ все равно выполняется.

### `combined_data.json` не найден

Файл должен лежать здесь:

```text
data/combined_data.json
```

Или нужно задать:

```powershell
$env:SOURCE_DATA_PATH = "путь_к_combined_data.json"
```

## Ограничения проекта

- Качество поиска зависит от качества и полноты `combined_data.json`.
- LLM может формулировать ответы немного по-разному.
- Структура сайта `priem.unecon.ru` может измениться, тогда парсер таблиц придется адаптировать.
- Бесплатный Render может засыпать.
- SQLite на Render не гарантирует постоянное хранение данных.

## Краткое резюме

Проект объединяет чат-бота приемной комиссии, парсер конкурсных списков и агента анализа ошибок. Важная часть диплома - не только ответ бота, но и проверка качества этого ответа: система сравнивает ответ агента с источником и эталонным ответом сотрудника, ищет релевантные фрагменты в базе источников и с помощью LLM формирует понятное объяснение ошибки.
